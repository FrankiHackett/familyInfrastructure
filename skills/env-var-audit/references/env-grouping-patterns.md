# Env Variable Grouping Patterns

How to produce a well-structured `.env.local.template` with logical service groups, inline documentation, and correct optional/required marking.

---

## Template Structure

```
{file header comment}

{group 1 header}
{group 1 variables}

{group 2 header}
{group 2 variables}

...

{test variables group — commented out}
```

---

## Group Order

Render groups in this order for readability:

1. **Supabase** — database, always first
2. **Anthropic** — primary LLM, if present
3. **OpenAI** — secondary LLM, if present
4. **Resend** — email, if present
5. **Stripe** — payments, if present
6. **Cloudflare** — infra, if present
7. **App Config** — app-specific values
8. **Test** — test-only vars, commented out by default
9. **Other** — unrecognised variables

---

## Full Template Example

```bash
# .env.local.template
# Generated by env-var-audit — {YYYY-MM-DD}
#
# Instructions:
#   1. cp .env.local.template .env.local
#   2. Fill in every blank value
#   3. Never commit .env.local (it's in .gitignore)
#   4. Re-run env-var-audit when adding new services
#
# Required vars marked with (required).
# Optional vars marked with (optional) and commented out by default.

# ── Supabase ─────────────────────────────────────────────────────────────────
# (required) Your Supabase project URL — find in: Project Settings → API
NEXT_PUBLIC_SUPABASE_URL=

# (required) Supabase anon key — safe for client-side use with RLS enabled
NEXT_PUBLIC_SUPABASE_ANON_KEY=

# (required) Supabase service role key — SERVER-SIDE ONLY. Never expose to browser.
SUPABASE_SERVICE_ROLE_KEY=

# (optional) Direct Postgres connection — only needed for migrations/scripts
# SUPABASE_DB_URL=

# ── Anthropic ─────────────────────────────────────────────────────────────────
# (required) Anthropic API key — SERVER-SIDE ONLY
ANTHROPIC_API_KEY=

# (optional) Override default model
# ANTHROPIC_MODEL=claude-sonnet-4-6

# ── Resend ───────────────────────────────────────────────────────────────────
# (required) Resend API key — SERVER-SIDE ONLY
RESEND_API_KEY=

# (required) Default from address for outbound emails
RESEND_FROM_ADDRESS=

# (optional) Display name for the from address
# RESEND_FROM_NAME=

# ── App Config ────────────────────────────────────────────────────────────────
# (required) Full public URL of this app (no trailing slash)
NEXT_PUBLIC_APP_URL=

# ── Test ─────────────────────────────────────────────────────────────────────
# Used in Playwright e2e tests only. Never used in production code.
# TEST_USER_EMAIL=
# TEST_USER_PASSWORD=
```

## Inline Documentation Rules

Every variable in the template must have an inline comment with:
1. `(required)` or `(optional)` marker
2. One-line description of what the value is
3. Where to find the value, if not obvious

```bash
# (required) Your Supabase project URL — find in: Project Settings → API
NEXT_PUBLIC_SUPABASE_URL=
```

For server-only variables, always add the safety note:

```bash
# (required) Anthropic API key — SERVER-SIDE ONLY
ANTHROPIC_API_KEY=
```

## Detecting Changes Between Runs

When the template already exists and a new audit is run, diff the new and old versions:

```bash
NEW_TEMPLATE="/tmp/env-template-new.txt"
EXISTING_TEMPLATE=".env.local.template"

# Generate new template to temp file first
generate_template > "$NEW_TEMPLATE"

if [ -f "$EXISTING_TEMPLATE" ]; then
  ADDED=$(diff <(grep -v '^#' "$EXISTING_TEMPLATE" | grep -v '^$' | sort) \
               <(grep -v '^#' "$NEW_TEMPLATE"      | grep -v '^$' | sort) | grep '^>' | sed 's/^> //')
  REMOVED=$(diff <(grep -v '^#' "$EXISTING_TEMPLATE" | grep -v '^$' | sort) \
                 <(grep -v '^#' "$NEW_TEMPLATE"      | grep -v '^$' | sort) | grep '^<' | sed 's/^< //')

  if [ -n "$ADDED" ]; then
    echo "NEW variables added to template:"
    echo "$ADDED"
  fi

  if [ -n "$REMOVED" ]; then
    echo "REMOVED variables (no longer used in code):"
    echo "$REMOVED"
  fi
fi

# Write the new template
cp "$NEW_TEMPLATE" "$EXISTING_TEMPLATE"
echo "Updated .env.local.template"
```

## validate-env.ts Variable List Format

The same grouping used for the template should be used when generating the `ENV_VARS` array in `validate-env.ts`:

```typescript
const ENV_VARS: EnvVar[] = [
  // ── Supabase ────────────────────────────────────────────────────
  { name: 'NEXT_PUBLIC_SUPABASE_URL',      required: true,  serverOnly: false, description: 'Supabase project URL' },
  { name: 'NEXT_PUBLIC_SUPABASE_ANON_KEY', required: true,  serverOnly: false, description: 'Supabase anon key' },
  { name: 'SUPABASE_SERVICE_ROLE_KEY',     required: true,  serverOnly: true,  description: 'Supabase service role key — server-side only' },

  // ── Anthropic ───────────────────────────────────────────────────
  { name: 'ANTHROPIC_API_KEY',             required: true,  serverOnly: true,  description: 'Anthropic API key' },

  // ── Resend ──────────────────────────────────────────────────────
  { name: 'RESEND_API_KEY',                required: true,  serverOnly: true,  description: 'Resend API key' },
  { name: 'RESEND_FROM_ADDRESS',           required: true,  serverOnly: true,  description: 'Default sender email address' },
]
```

The `serverOnly: true` flag controls whether the variable is validated on the server side only (not in browser bundles), preventing false validation errors for variables that legitimately don't exist in the browser environment.

## .gitignore Minimum Requirements

Every Household App Infrastructure app repo must have these entries:

```gitignore
# Environment variables — never commit
.env.local
.env.*.local
.env.production

# Bootstrap config
.bootstrap-config.json

# Personal data allowlist (if used)
.personal-data-allowlist
```

The template file (`.env.local.template`) **is** committed — it documents what values are needed without exposing them.
